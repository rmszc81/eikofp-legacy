Introdução

Olá amigo(o), seja bem vindo(a) ao Eiko Finanças Pessoais. Nesta página você encontrará algumas informações interessantes a respeito do que é esta aplicação, bem como as razões de ter sido desenvolvida e motivos de hoje estar sendo disponibilizada gratuitamente na internet como free-software (gratuito).

Um breve histórico...

O "Eiko Finanças Pessoais" é uma aplicação simples, destinada ao gerenciamento de pequenas finanças, ou seja, com poucos recursos disponíveis. No início, este projeto surgiu como uma brincadeira, um passatempo. Eu trabalhava numa fábrica de software na época e, no ínicio daquele projeto, nós desenvolvedores ficávamos muito tempo ociosos, por não ter realmente atividade para nós naquele dado momento. E como eu não gosto de ficar sem ter o que fazer, comecei a pensar no que eu poderia fazer para me ocupar.

Depois de alguns minutos pensando, pensei:
- "Porque não procurar uma aplicação para fazer um gerenciamento de minhas finanças?"

Comecei então, a procurar por aplicações de finanças gratuitas e conforme fui encontrando, fui baixando, instalando, testando e analisando uma a uma, para encontrar aquela que se encaixasse na minha expectativa. No final das contas, após uma semana inteira nisso, perdi as esperanças de encontrar algo que eu esperava, ou seja, simples. Todas eram aplicações muito complicadas, com muitos detalhes e dotadas de parafernálias infindáveis de recursos que eu tinha certeza absoluta que jamais iria utilizar.

Depois de uns dias, ao olhar o extrato bancário, pensei:
- "Porque não desenvolver uma aplicação pra mim, do meu jeito?"

Foi aí que começou a brincadeira. Trabalhando como desenvolvedor numa fábrica de software, comecei a pegar uma idéia aqui, outra ali e assim a aplicação nasceu. Foi crescendo pouco a pouco e os recursos foram implementados conforme eu fosse precisando. Depois de um bom tempo desenvolvendo, a aplicação já estava pronta para entrar em fase de testes. Entretanto, na época, devido à falta de tempo, os testes nunca foram realizados e aplicação ficou abandonada por um tempo.

No início (e atualmente) a aplicação foi construída em Visual Basic 6 ™. O banco de dados utilizado na época era o Access ™, ou seja, arquivos MDB. Porém, conforme a aplicação foi crescendo, os arquivos MDB se tornaram inviáveis e aí, surgiu um problema. O banco de dados obrigatoriamente, teria de ser mudado. E uma vez que o banco de dados fosse mudado, uma boa parte da aplicação teria que ser reescrita.

Alguns dias depois, escolhi o SQL Server ™ 2005 Express. Gratuito porém, mais pesado que o Access ™, infelizmente. Depois de um tempo reescrevendo o que era necessário para a aplicação rodar com o novo banco de dados, importei os dados da base antiga e iniciei a fase de testes. Obviamente erros apareceram e foram corrigidos pouco a pouco. O meu tempo era escasso na época e isso ocasionou uma demora maior para que a aplicação entrasse em fase de produção.

Foi naquele momento que comecei a utilizar meu próprio programa. A versão da época então continha algumas funcionalidades, dentre as quais eram:

- Cadastro de contas
- Cadastro de receitas
- Cadastro de despesas
- Cadastro de formas de pagamento
- Contas a pagar
- Contas a receber
- Agenda financeira
- Movimentação

Não havia até então autenticação por usuário e senha na aplicação. Qualquer um que operasse a máquina teria acesso às minhas finanças. Era só abrir o programa e tudo estava lá, na tela para ser visualizado. E como eu não queria que isso acontecesse, posteriormente, foi implementado um cadastro de usuário simples. Este obrigava o usuário a se autenticar antes de ter acesso aos dados.

Surgiu então, um outro problema. A aplicação não tinha sido construída para suportar vários usuários. Mesmo com o cadastro de usuários implementado, todos tinham acesso aos mesmos dados. Obviamente, boa parte da aplicação teve de ser reescrita e a estrutura da base de dados também foi modificada. Depois de tudo pronto, mais uma pequena fase de testes e aparentemente, tudo estava funcionando perfeitamente bem.

Muuuito tempo depois, um conhecido sugeriu implementar gráficos na aplicação. Então, um gráfico foi implementado como piloto, para vermos como ficava. Ambos gostamos do resultado e o mesmo gráfico passou a fazer parte da aplicação. Surgiu então, outra necessidade. Eu precisava de um comparativo, de receitas x despesas em um determinado período, para ter uma idéia melhor de como eu estava movimentando o meu dinheiro. Essa solução então, foi implementada, testada, gostei do resultado e agora faz parte da aplicação também. :)

O programa estava rodando legal, tudo estava funcionando muito bem e não havia problemas conhecidos até então. Posteriormente, foi iniciada uma nova fase de produção. Com o tempo, fui mostrando a aplicação para algumas pessoas. Algumas se ofereceram para comprá-la pois acharam uma solução bastante interessante, mesmo sendo relativamente simples. Eis então que surgiu o maior de todos os problemas até o momento: Distribuir a aplicação. Mas, se tudo estava perfeito, qual era o problema?

Simples, simples, simples. O banco de dados. :(

Naquela altura, a aplicação criava automaticamente a base de dados, as tabelas e tudo mais que fosse necessário para rodar no SQL Server ™ 2005 Express. Mas, se a aplicação criava tudo sozinha, qual era o problema afinal?? Instalar o servidor de banco de dados e configurá-lo de forma correta. Para quem é desenvolvedor como eu, isso é uma tarefa das mais simples, porém, para um usuário que nunca viu isso na vida, seria uma tarefa bastante complexa.

Infelizmente, devido a tudo isso, fui obrigado a dizer a estas pessoas que, naquele momento, não era possível distribuir a aplicação. Mesmo sendo sincero e explicando a real situação, muitas perderam o interesse e desistiram de utilizá-la. Até aí, muito mais do que óbvio. Como vou usar uma aplicação se é difícil até para instalar? Eis que surgiu um novo desafio, como fazer um pacote de instalação simplificado?

Procurei durante algum tempo uma solução para isso. Encontrei? Sim, porém, era totalmente inviável, pois a aplicação que era simples, ao ser instalada junto com o banco de dados, chegava a ter mais de 60 MB só o instalador! E convenhamos, para uma aplicação desse porte, 60 MB é exagero demais da conta. Continuei então utilizando minha aplicação e deixei de lado a idéia de distribuí-la durante um tempo.

Alguns meses depois, conheci um banco de dados que nem sabia de sua existência, o SQLite ™. E o melhor? É grátis. Fui até o site do projeto, baixei os arquivos necessários, fiz vários testes e obtive resultados bastante satisfatórios. Todos os recursos que eu utilizava no SQL Server ™ eram então suportados pelo SQLite ™, além de não ter os mesmos problemas que enfrentei na época com o Access ™.

Legal, gostei. Mas e aí? E aí que surgiu um outro problema. :(

Até então, não havia nenhuma biblioteca compatível com Visual Basic 6 ™ para acesso ao SQLite ™. Alguns meses depois, quando eu já havia desistido da idéia totalmente, numa madrugada fria, eu estava sem sono, levantei e resolvi vir até o micro dar uma navegada na internet. Para minha surpresa, sem querer, achei um site de um projeto que havia criado uma biblioteca que pudesse ser utilizada com o Visual Basic 6 ™ para acesso ao SQLite e ainda, gratuita! Yes, yes, yes! :D

Imediatamente eu fiz o download do conteúdo necessário, inclusive peguei até alguns exemplos de como usar a biblioteca. Na manhã seguinte, após acordar e tomar aquele delicioso café da manhã, resolvi dar uma olhada no que tinha achado, para ver se era realmente o que eu precisava e bingo! Tinha tudo o que eu precisava e mais um pouco. Peguei os exemplos, fiz uma verdadeira bateria de testes. Realmente, agora era possível usar o SQLite ™ na minha aplicação.

Decidi que, não iria somente portar a aplicação para o novo banco de dados. Eu queria criar algo novo, estável, porém, sem alterar o visual que já tinha criado. E assim foi durante aproximadamente dois meses. A aplicação foi quase que totalmente reescrita e hoje ela inclui várias funcionalidades novas que, a princípio, nunca existiriam pois não eram necessárias com o SQL Server ™.

Para testar, importei a minha base de dados do SQL Server ™ para o SQLite ™. Devido à quantia de dados ser relativamente considerável, serviu também para realizar testes de performance. Além de estar mais leve, a aplicação também está muito mais rápida do que era antes. Toda a aplicação foi testada novamente. Todos os testes foram satisfatórios e os erros que foram encontrados foram corrigidos. Mesmo após a correção dos erros, todos os testes foram refeitos, para certificar-me de que tudo está funcionando perfeitamente.

E para esta versão, foi incluído também um arquivo de ajuda. Neste arquivo você encontrará explicações de como funciona a aplicação bem de como utilizar seus recursos. Há também um tutorial, caso tenha dúvidas, não deixe de conferir.

A partir daqui, aceito sugestões de melhoria e correção, com as seguintes ressalvas:

- Correções terão prioridade maior que as melhorias.
- Melhorias serão analisadas, tanto tecnicamente, quando em questões de viabilidade.
- As tarefas serão feitas conforme a minha disponibilidade de tempo.

E finalizando (finalmente, ufa), futuramente a aplicação será totalmente reconstruída na plataforma .NET ™ ou ainda, em JAVA ™. Até o momento eu não decidi qual será, porém, até lá eu já terei encontrado a melhor solução para isso. Os motivos da reconstrução incluem principalmente pelo Visual Basic 6 ™ já estar muito ultrapassado hoje em dia. E como o mundo evolui, temos de evoluir também. Todo mundo sabe que, quem anda sempre no mesmo ritmo, cedo ou tarde, acaba ficando pra trás. Enfim, é isso.

Espero que goste da aplicação.
E se não gostar, é simples, é só não usar.
